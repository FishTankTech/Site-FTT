FTT Public Site InfrastructureThis repository hosts the static assets, configuration, and documentation necessary to build and host the Fish Tank Tech (FTT) public-facing website.Purpose and PhilosophyIn line with the FTT core philosophy, this repository is treated as infrastructure, not a showcase or a product. Its primary goals are legibility and durability.Low Dependency: The codebase minimizes external dependencies to ensure long-term stability and reduce the maintenance surface area.Clear Intent: The structure and code prioritize simplicity and explainability. A competent non-developer should be able to navigate the repository and understand how the site is assembled.Content-First: The site's primary function is to deliver clear, technical context. The repository structure is optimized for managing and deploying this static, archival content.Explicit Constraints (Why it is simple)This repository is primarily a deployment vehicle and content container.No Complex Logic: The site is static. There is no server-side business logic, database, or API integrations managed within this repository.Minimal Tooling: Build processes are deliberately simple and documented to reduce the "black box" effect.Zero Hype: Design and functionality choices favor clarity over trend-driven aesthetics or unnecessary front-end complexity.Structure OverviewDirectoryPurposeNotes/content/Site source files (Markdown, etc.)The core of the FTT narrative and documentation./layouts/Templating and structureDefines the minimalist presentation layer./static/Static assets (CSS, images)Low-level styling focused on legibility and hierarchy./config/Build configurationExplicit configuration for static site generation.Maintenance & ModificationFuture modifications should uphold the FTT principle: Maintainability is a feature, not an afterthought.To Modify Content: Edit files within the /content/ directory.To Modify Presentation: Adjust layouts and minimal CSS within /layouts/ and /static/.To Extend Functionality: Any proposed additions must be weighed heavily against the FTT goal of reducing ambiguity and dependencies. New dependencies require explicit, documented justification.Assumption: The reader wants to know how things work, and may eventually need to fork or extend this simple structure for their own documentation systems.
